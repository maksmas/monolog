# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Monolog is a CLI personal backlog tool written in Go. Tasks are stored as individual JSON files in a git repo for conflict-free cross-device sync. See `docs/plans/completed/20260411-monolog-cli-mvp.md` for the MVP spec.

## Build & Test Commands

```bash
go build -o monolog          # Build binary
go test ./...                # Run all tests
go test ./internal/store/    # Run tests for a single package
go test -run TestCreate ./internal/store/  # Run a single test
go vet ./...                 # Lint
```

## Architecture

```
main.go                 → Entrypoint, calls cobra root command
cmd/                    → One file per CLI command (add.go, ls.go, etc.)
internal/config/        → User-facing date-format and theme accessors; Load/Save/SetDateFormat for settings persistence
internal/model/task.go  → Task struct with JSON tags, ULID generation
internal/store/         → File-based CRUD, prefix-match ID lookup, initials resolution, filtering/sorting
internal/ordering/      → Fractional position math and rebalancing
internal/schedule/      → Bucket names ↔ ISO dates, bucket classification, layout-aware Parse/FormatDisplay
internal/recurrence/    → Recurrence-rule parsing, next-occurrence math, shared done+spawn flow
internal/display/       → Terminal table formatting and compact date helpers (layout-aware)
internal/git/           → Git operations (init, auto-commit, sync with conflict resolution); also AutoCommitSHA (stage+commit+return HEAD SHA), headSHA, CommitSubject, Revert, RevertSHA helpers used by TUI undo/redo
internal/tui/           → Bubble Tea interactive TUI (default when run with no subcommand)
```

Commands in `cmd/` handle CLI parsing and delegate to `internal/` packages. Running `monolog` with no subcommand launches the interactive TUI. Every mutation auto-commits to git.

## Key Design Decisions

- **One JSON file per task** (`<repo>/.monolog/tasks/<ULID>.json`): eliminates merge conflicts on sync. Two devices editing different tasks rebase cleanly.
- **ULID IDs**: time-sortable, globally unique, users type partial prefixes (e.g. `01J5K`) which resolve via directory scan.
- **Title-initials lookup**: CLI commands (`done`, `edit`, `rm`, `mv`) resolve identifiers via `store.Resolve()`, which tries ULID prefix first, then falls back to matching the first letters of each word in open task titles (e.g. `flb` → "Fix login bug"). Minimum 2 characters, case-insensitive, prefix-matching on initials. Ambiguous matches list conflicting titles. Implemented via `model.Initials()` and `store.Resolve()`.
- **Fractional positions**: inserting between positions 1000 and 2000 gives 1500. Rebalance all positions in a schedule group when any gap drops below 1.0. Default spacing is 1000.
- **Schedule values**: `today` (default), `tomorrow`, `week`, `month`, `someday`, or a date. User-facing input/display format is configurable via `internal/config` (`DateFormat()`, `DateFormatLabel()`, `DateRegex()`); default is `DD-MM-YYYY` (`"02-01-2006"`). `schedule.Parse(input, now, layout)` accepts bucket names, input in the passed layout, and legacy ISO (`2006-01-02`) silently so pre-existing scripts keep working. `schedule.FormatDisplay(iso, layout)` renders a stored ISO date back to the configured layout at display time. On-disk `schedule` JSON stays ISO regardless of `DateFormat` — no migration when the display format changes. Callers pass `config.DateFormat()` through to `schedule`/`display`/`model` helpers; those packages never import `internal/config` directly, which keeps them easy to test with arbitrary layouts. **Relative shorthands** (`Nd`, `Nw`, `Nm` — e.g. `3d`, `2w`, `1m`) are handled by `schedule.ParseRelative(input string, now time.Time) (string, bool)`, which the TUI `updateReschedule` calls before `schedule.Parse`; on a match it returns an ISO date directly without touching `schedule.Parse`'s signature or behavior.
- **Runtime data directory**: `~/.monolog/` by default, override with `MONOLOG_DIR` env var.
- **Active tasks**: the reserved `active` tag marks tasks in the current working set. Active tasks render in green in the TUI and appear in a dedicated panel above the tab bar. `done` auto-deactivates; `edit --tags` preserves active state.
- **Auto-tag from title prefix**: when creating a task, if the title starts with `"tagname: ..."` and `tagname` is a tag that already exists on another task, it is automatically added as a tag. Title is kept as-is. The reserved `active` tag is excluded. Implemented via `model.ParseTitleTag()` and `model.CollectTags()`.
- **Tag view mode**: the TUI has two view modes (`viewSchedule` / `viewTag`), toggled with `v`. In tag view, tabs become tags (active first, alphabetical middle, untagged last), each showing tasks grouped by schedule bucket with separator items. Grab mode in tag view restricts movement to within a single tag tab (no left/right). Launch directly into tag view with `monolog --tags` / `-T`.
- **CompletedAt lifecycle**: `Task.CompletedAt` (RFC3339, omitempty) is set when `doneSelected()` fires. It is never cleared (tasks cannot be reopened today). Existing tasks without `CompletedAt` are excluded from avg-time-to-complete calculations. Used by `model.ComputeStats()`.
- **Stats bar**: the TUI always shows a one-line stats bar (total/open/done/in-tab/avg) above the tab bar. Stats are cached in `Model.stats` (a `model.Stats` value) and refreshed via `reloadAllTasks()` at the end of every `reloadAll()` call. `model.ComputeStats([]Task, time.Time)` and `model.FormatDuration(days float64)` in `internal/model/stats.go` are pure functions, easily unit-tested.
- **Tag autocomplete**: both the add-task and retag modals show inline autocomplete suggestions as the user types in the tag field. `model.FilterTags(known, field)` is a pure function that extracts the fragment after the last comma, filters known tags by case-insensitive prefix, excludes already-entered tags, and caps at 5 results. Suggestions are navigated with Up/Down, accepted with Tab/Enter (replaces fragment and appends `", "`), and dismissed with Esc. State lives in `Model.suggestions`/`Model.suggestionIdx`, shared between modals since only one is open at a time. The add modal's Recur field reuses the same suggestion plumbing via the pure `recurrence.Suggest(input)` function and shows a static grammar hint line (`recurrence.GrammarHint`) under the input — on the Recur field only **Tab** accepts a suggestion (replacing the full value, since recurrence is a single value) while **Enter** always submits the modal; the same `GrammarHint` is also prepended as a `#` comment header in the TUI YAML edit buffer and referenced in the `--recur` CLI help on `add`/`edit`.
- **Task notes**: notes are stored as timestamped entries appended to the existing `Body` field using `--- <date> HH:MM:SS ---` separators, where `<date>` is rendered in the configured `config.DateFormat()` (default `DD-MM-YYYY`) — no schema migration needed. A `NoteCount int` field (JSON `note_count`, omitempty) tracks the count for fast badge rendering (`[N]` in the task list) without body parsing. `model.AppendNote(body, text, now, layout)` formats and appends note text; `model.CountNotes(body, dateRegex)` only recognizes separators whose date fragment matches the passed regex. Callers build the regex from `config.DateRegex()`. **No backward compat** on the separator regex: changing the configured format loses the `[N]` count contribution from notes written under the old format (stored text is untouched). `NoteCount` is recalculated from `Body` inside both `Store.Create` and `Store.Update` via `model.CountNotes()` — the store layer is the single source of truth, so any path that writes a task (new or existing, including recurring-task spawns whose body already carries the `Spawned from` note) keeps the count consistent without caller bookkeeping. CLI: `monolog note <id> "text"` adds a note, `monolog show <id>` prints full task detail.
- **Detail panel**: pressing `Enter` in the TUI toggles a right-side panel showing task metadata (title, schedule, tags, dates) and the full body with notes. An inline textarea at the bottom accepts new notes (`Enter` submits, `Alt+Enter` inserts a newline, `Esc` closes the panel). Navigating tasks with `Up/Down` while the panel is open refreshes its content. Panel width is ~45% of the terminal.
- **Fuzzy search**: `/` in TUI normal mode opens a full-screen search overlay that fuzzy-matches across both title and body of all tasks (open and done), powered by `github.com/sahilm/fuzzy`. Title matches are weighted 2x over body matches. Enter commits to the selected task, switching `activeTab` to the target bucket/tag and focusing the task; Esc cancels with no side effects. Pure ranker lives in `internal/tui/search_match.go`; overlay UI in `internal/tui/search.go`.
- **Recurring tasks**: `Task.Recurrence string` (JSON `recurrence,omitempty`) holds a rule in one of four forms — `monthly:N` (1..31, clamped to month-end for short months), `weekly:<day>` (accepts three-letter `mon|tue|...|sun`, full names `monday|...|sunday`, or numeric `1..7` with Mon=1, all case-insensitive), `workdays` (Mon-Fri), or `days:N` (N days after completion). Parsing and next-date math live in `internal/recurrence/` (`Parse(string) (Rule, error)`, `Canonicalize(string) (string, error)`, `Rule.Next(time.Time) time.Time`, `Rule.String()`); `Rule.String()` emits the canonical form (`weekly:mon`, `workdays`) so alias inputs round-trip through edit to canonical storage. All three callers (`cmd/add`, `cmd/edit`, and the TUI add modal / YAML editor) validate-and-normalize through `recurrence.Canonicalize`. Spawn is driven by `recurrence.CompleteAndSpawn(s *store.Store, task *model.Task, now time.Time, w io.Writer, dateFormat string)` (shared by `cmd/done.go` and the TUI `d` key), where `w` receives any spawn-warning messages and `dateFormat` is the configured user-facing layout passed in from `config.DateFormat()`. When a completed task has a non-empty `Recurrence`, a new task is created with a fresh ULID, copied Title/Body/Source/Recurrence, tags minus `active`, `Schedule = Rule.Next(now)` (stored ISO); the old task receives the done-state transition AND the back-reference note in a single `Store.Update`, and both files are committed in one git commit whose message formats the next date with `dateFormat` (e.g. `"done: <title> (recurring, next 25-04-2026)"` under the default). Bidirectional cross-references: the new task gets a `"Spawned from <old-ULID>"` note, the old task gets a `"Spawned follow-up: <new-ULID> (scheduled <date>)"` note formatted in `dateFormat`. Stop semantics are edit-based — clearing `Recurrence` (CLI `--recur ""` or removing the `recurrence:` line in the TUI YAML editor, which now exposes it through `editableFields`) before completing prevents the next spawn; there is no SeriesID or series concept. Any spawn failure (invalid stored rule, list/create error) is warned to the caller's stderr and skipped while the underlying completion still succeeds — the repo is never left with an uncommitted half-spawn. CLI surface: `--recur` flag on `add` and `edit`; TUI surface: a third input `addFocusRecur` in the add modal and a `recurrence` line in the YAML editor.
- **Clickable URLs (TUI)**: the TUI wraps `http://` and `https://` URLs in OSC 8 terminal hyperlink escapes so Cmd/Ctrl-click on a visible URL opens the browser. The pure `display.Linkify(s string) string` helper (`internal/display/links.go`) detects URLs with `https?://\S+`, strips trailing punctuation (`.,;:!?)`) so sentence punctuation stays outside the link, and wraps each match as `\x1b]8;;<URL>\x1b\\<URL>\x1b]8;;\x1b\\`. OSC 8 sequences are zero-width so `lipgloss` column math is unaffected, and styling composes cleanly. Integration points — always applied post-wrap / post-truncate on the final visible fragment so escapes never dangle across a cut: the detail-panel title header, the detail-panel body lines, and the task-list title column. **Contract for new render paths**: any caller applying `display.Linkify` to width-constrained output must first pass the text through the URL-aware pre-processors (`wrapTextPreservingURLs` for wrapping, `truncateTitlePreservingURLs` for truncation, both in `internal/tui/model.go`; `display.FindURLSpans` for custom layout logic) so a URL is never split across the visible fragment — otherwise Linkify's regex would see a truncated match and the OSC 8 target would point at a broken URL. `vlist.itemHeight` must mirror whichever wrap helper the renderer uses so row-height prediction stays aligned with rendered output. `MONOLOG_NO_LINKS=1` is an escape hatch that turns `Linkify` into a no-op (also covered by tests via `t.Setenv`). Terminals without OSC 8 support render URLs as plain text — no regression. CLI output (`monolog show`, `monolog ls`), the fuzzy-search overlay, and the YAML editor intentionally stay plain text.
- **TUI color themes**: named color themes ("default" and "dracula") are selectable via the `"theme"` key in `<MONOLOG_DIR>/.monolog/config.json`; the `MONOLOG_THEME` env var takes precedence over the file. `Theme` struct in `internal/tui/theme.go` holds one `lipgloss.AdaptiveColor` field per color role (chrome: `ActiveBorder`, `ModalBorder`, `Hotkey`, `ActiveTabBg`, `ActiveTabFg`, `TabFg`, `Borders`; task list: `NormalText`, `SelectedText`, `DimText`, `GrabText`, `ActiveNormal`, `ActiveSelected`; search: `SearchDone`, `SearchActive`, `SearchCount`, `SearchMeta`, `SearchPreviewBorder`, `SearchPreviewDim`); the `themes` registry maps names to `Theme` values and unknown names silently fall back to `defaultTheme`. `config.Theme()` in `internal/config/` reads the JSON config file and applies the env-var override. `buildStyles(Theme) tuiStyles` in `tui.go` constructs all lipgloss styles from theme fields (replacing the former package-level `var` block); `initStyles(Theme)` in `model.go` handles task-list item styles; `Model` gains `theme` and `styles` fields used for inline color overrides throughout the render paths. Theme-independent styles (`suggestionBoldStyle`, `suggestionDimStyle`, `searchSelectedStyle`) live as package-level vars in `tui.go`. Both `"theme"` and `"date_format": "02-01-2006"` keys are written by `git.Init` into the initial `config.json` template. **User-authored themes (file-based)**: `internal/tui/theme_load.go` scans `<MONOLOG_DIR>/.monolog/themes/*.json` at startup via `loadUserThemes` and merges them into the package-level `themes` map through `initThemes` — both called from `tui.Run` before the Model is built. Files use snake_case keys (`active_border`, `normal_text`, …) with `{"light": "...", "dark": "..."}` pairs; all 19 fields are required. `bootstrapExampleTheme` creates the themes directory with a single `example.json` (a full dump of `defaultTheme`) the first time the directory is absent — it never overwrites or recreates once the directory exists, so deleting `example.json` is a sticky user choice. Loader rejections (invalid JSON, missing field, empty color value) are printed to stderr with a `monolog:` prefix before Bubble Tea takes over; the TUI continues with whatever loaded cleanly. User files named `default.json` or `dracula.json` are rejected as collisions — built-ins are immutable. If the configured theme name disappears from disk between runs, `newModel` falls back to `defaultTheme` and surfaces `theme "<name>" not found, using default` in the status bar without rewriting `config.json` (restoring the file later brings the selection back automatically). Reload is edit-and-restart — no file watcher.
- **Settings modal**: `,` in normal TUI mode opens a settings overlay (`modeSettings`) backed by `openSettings()` / `updateSettings()` / `applySettings()` in `model.go`. Two rows: date format (DD-MM-YYYY / MM-DD-YYYY / YYYY-MM-DD) and theme. `↑↓` navigate rows; `←→` cycle values; `Enter` saves via `config.SetDateFormat` + `config.Save(repoPath, theme, fmt)` and applies the new theme live by rebuilding `m.theme`/`m.styles`/`m.baseStyles`; `Esc` discards. `config.Load(monologDir string) error` is called in `cmd/helpers.go#openStore` so every command (TUI + CLI) picks up the persisted date format on startup. `config.AllFormats() []FormatInfo` (Layout + Label fields) exposes the ordered format list to the settings UI. `config.Save` does a read-modify-write so unknown config.json keys (e.g. `"default_schedule"`, `"editor"`) are preserved.
- **Undo (TUI)**: `u` / `ctrl+z` in normal mode triggers multi-level undo (last 10 actions) backed by `git revert <sha> --no-edit`. TUI mutations use `git.AutoCommitSHA` (stage+commit+return HEAD SHA) while CLI commands use `git.AutoCommit` (no SHA returned). Every TUI mutation captures the SHA into `Model.undoStack` ([]string, capped at 10, newest last). `undoCmd()` **pops the SHA before launching the goroutine** (prevents a second keypress racing on the same SHA), then calls `git.CommitSubject` to build the status message and `git.RevertSHA` (thin wrapper over `git.Revert` + `headSHA`) to create the revert commit and capture its SHA. `CommitSubject` failure is treated as non-retriable (stale SHA — e.g. after a rebase) and the SHA is dropped. `RevertSHA` failure (e.g. conflict) signals restore via `taskSavedMsg.restoreUndoSHA` so `Update` pushes the SHA back in the main loop — never from inside the goroutine (which would race). Stack is cleared when a sync with a remote completes (rebase may have rewritten SHAs); local-only syncs preserve the stack. Grab-move rebalance positions are not captured in the move commit and thus not reverted by undo (pre-existing limitation of the move commit).
- **Redo (TUI)**: `ctrl+y` in normal mode re-applies the last undone action (multi-level, same 10-entry cap as undo) by calling `git.RevertSHA` on the revert commit from the prior undo — reverting the revert re-applies the original change. `Model.redoStack` ([]string, capped at `undoStackCap`) mirrors `undoStack`: each successful undo pushes the revert commit's SHA onto `redoStack` (via `taskSavedMsg.redoSHA`); `redoCmd()` pops and reverts that SHA, then pushes the new SHA back onto `undoStack` via `taskSavedMsg.redoneSHA` (which preserves `redoStack`, unlike the plain `sha` field) so `u` → `ctrl+y` → `u` cycles symmetrically. `restoreRedoSHA` mirrors `restoreUndoSHA`: on redo failure the SHA is pushed back onto `redoStack` from the main loop. Standard redo semantics: any new mutation (non-empty `taskSavedMsg.sha`) clears `redoStack`, since the prior redo history is no longer valid. The former `clearUndo` flag is renamed to `clearHistory` and clears **both** stacks — used by remote sync, since a rebase may rewrite SHAs referenced in either stack. Empty-stack guard in `updateNormal` shows "nothing to redo" without calling the goroutine.

## Dependencies

Minimal: cobra (CLI framework), oklog/ulid/v2 (ID generation), charmbracelet stack (bubbletea, bubbles, lipgloss for TUI; charmbracelet/x/term for TTY detection in the pager helper), sahilm/fuzzy (TUI fuzzy search ranker), yaml.v3 (YAML editor round-trip in TUI edit).
